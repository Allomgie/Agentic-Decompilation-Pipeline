IDO Agent Pipeline

Eine agentische Pipeline, die handgeschriebene und maschinell erzeugte C-Entwürfe automatisiert auf einen byte-exakten Match gegen echte N64-MIPS-Assembly treibt — mit deterministischer Reparatur, einem lokalen LLM, einem Permuter und einer selbst aufgebauten Ähnlichkeits-Wissensbasis.
IDO 5.3 cc MIPS / -O2 -G0 Qwen3 / Gemma4 (vLLM) m2c decomp-permuter WSL / Python 3.12 Banjo-Tooie decomp

0TL;DR

Das Ziel der Banjo-Tooie-Dekompilierung ist, für jede Funktion C-Code zu finden, den der originale IDO-5.3-Compiler in exakt dieselben Bytes übersetzt wie das Spiel-ROM. Diese Pipeline automatisiert genau diese Suche.

Sie besteht aus zwei eigenständigen Wrappern, die sich eine gemeinsame Modul-Bibliothek (Compiler-Treiber, Match-Bewertung, LLM-Schnittstelle, Permuter, Wissensbasis) teilen:

0_wrapper.py — Hauptpipeline Die erprobte Produktions-Pipeline für normale Funktionen. Nimmt einen C-Entwurf, repariert ihn deterministisch + per KI bis er kompiliert, und minimiert dann iterativ den ASM-Diff zum Ziel, bis 100 % Bytes stimmen. Läuft über tausende Funktionen parallel.
1_wrapper_mother.py — Mutter-Wrapper Spezial-Pipeline für große Funktionen. Die Hauptpipeline liefert ihre besten Ergebnisse auf kleinen Funktionen; große „Mütter“ werden hier daher kontextbewusst in kleinere Häppchen („Arme“) serviert. Baut auf dem Alt-Splitter auf: kompiliert im Mutter-Kontext (PIC) und matcht Arm für Arm front-to-back auf byte-exakt.
Was diese Pipeline besonders macht
  • Wasserdichtes Match-Kriterium: Vergleich zweier unrelocierter .o-Dateien (Hex + Relocation-Symbol/Typ) statt naivem .s-Textvergleich.
  • Deterministik vor KI: Auto-Fixes, Cluster-Transplantate und ein Permuter lösen viele Fälle mit 0 KI-Tokens.
  • Selbstlernend: jeder Perfect Match speist eine Ähnlichkeits-DB (14 641 Einträge) und ein Pattern-Cheatsheet, die künftige Übersetzungen leiten.
  • Adaptive Eskalation: oneshot → thinking → best-of-N-Batch, gesteuert durch echten Match-Fortschritt, nicht durch feste Budgets.

★Agentic-AI auf einen Blick

Diese Pipeline ist ein autonomer Agent mit verifizierbarem Ziel: er arbeitet ein hartes, objektiv messbares Erfolgskriterium (byte-exakter Match) in einer geschlossenen Wahrnehmungs-/Handlungs-Schleife ab — und setzt das LLM gezielt dort ein, wo es einen Vorteil bringt, statt blind zu prompten.

Agentic-KonzeptUmsetzung in dieser Pipeline
Closed-Loop-Agent (sense→plan→act) kompilieren → mit Ziel vergleichen → JSON-Diff interpretieren → C gezielt ändern → wiederholen, bis 100 % Bytes stimmen (§4.1).
Verifizierbares Reward-Signal byte-exakter .o-vs-.o-Vergleich statt „klingt plausibel“ — jeder Schritt ist objektiv bewertbar (§3).
Werkzeuggebrauch (tool use) der Agent steuert echte Tools: IDO-Compiler, objdump, m2c, Hill-Climbing-Permuter, Ähnlichkeits-Suche.
Adaptive Eskalation & Budget-Steuerung oneshot → thinking → best-of-N-Batch, ausgelöst durch echten Fortschritt, nicht durch feste Budgets (§4.2).
Determinismus vor LLM Auto-Fixes, Cluster-Transplantate und Permuter lösen viele Fälle mit 0 Tokens; das Modell läuft nur, wo es nötig ist (§4.4).
Lernen während der Ausführung (RAG / Memory) jeder Erfolg speist Ähnlichkeits-DB + Cheatsheet → folgende Funktionen starten besser, bis die kritische Masse erreicht ist (§6).
Aufgaben-Zerlegung (task decomposition) der Mutter-Wrapper zerlegt große Funktionen kontextbewusst in kleine Arme und matcht sie einzeln front-to-back (§5).
Kontext-Engineering der Kontext-Fahrplan staffelt, was das Modell sieht; Reasoning wird nie in die nächste Nachricht zurückgespielt (§4.3).
Persistenz & Wiederaufnahme Zustand pro Funktion überlebt Neustarts; --resume setzt Läufe fort (§7).
Modell-Portabilität & Kosten dasselbe Agenten-Verhalten läuft auf lokalem GPU-LLM oder Cloud-API — bewusst zur Kostenkontrolle (§4.5).
Der entscheidende Effekt Die Pipeline wird während sie läuft besser: gelöste Funktionen werden zu Referenzen für die nächsten. Ab einer kritischen Masse lösen sich ähnliche Funktionen nahezu von selbst — ein selbstverstärkender Agenten-Kreislauf statt eines statischen Skripts.

1Problem & Kontext

Eine Dekompilierung (decomp) rekonstruiert den C-Quelltext eines kompilierten Spiels so, dass er bitgenau dieselbe Binary erzeugt. Für N64-Spiele wie Banjo-Tooie heißt das: der historische SGI IDO 5.3 cc muss aus dem Kandidaten-C exakt die Bytes des ROMs produzieren. Erst dann gilt eine Funktion als „matched“.

Warum das schwer ist Es reicht nicht, semantisch korrekten C zu schreiben. Registerwahl, Instruktions- Reihenfolge, Stack-Layout und Optimierungs-Eigenheiten müssen exakt dem entsprechen, was IDO 5.3 vor 25 Jahren erzeugt hätte. Schon eine andere Klammerung, ein falscher Typ oder eine umsortierte Anweisung erzeugt andere Bytes. Tausende Funktionen von Hand so zu treffen ist extrem mühsam.

Die Pipeline automatisiert diese Schleife: kompilieren → mit dem Ziel vergleichen → den Diff verstehen → den C gezielt ändern → wiederholen — mit einem LLM als „Übersetzer“ und einer Reihe deterministischer Werkzeuge, die dem Modell die Arbeit abnehmen, wo Determinismus verlässlicher ist.

3Das Match-Kriterium — modules/ido_comparison.py

Das Herz jeder Bewertung ist die Frage: Sind die kompilierten Bytes byte-identisch zum Ziel? Naiv würde man den disassemblierten .s-Text vergleichen — das ist aber falsch, weil das ROM-.s bereits aufgelöste Relocations enthält (jede %hi/%lo/jal-Adresse), die ein frisch kompiliertes Objekt noch nicht hat.

Wasserdichter .o-vs-.o-Vergleich Die Pipeline assembliert die Ziel-.s (mit prelude.inc) einmalig zu einer unrelocierten .o und vergleicht sie mit der Kandidaten-.o: rohe Immediates sind beidseitig 0, und Relocations werden über Symbol + Typ (R_MIPS_*) verglichen — nicht über bereits aufgelösten Hex. Das Ziel-.o wird pro Pfad gecacht.

Daraus berechnet evaluate_match zwei Kennzahlen:

MetrikBedeutung
match_rateAnteil byte-identischer Instruktionen inkl. korrekter Relocations. 100 % = perfekter Match.
struct_scoreStruktureller Score (SequenceMatcher auf einem maskierten „Skeleton“): robuster gegen Relocations. struct 100 % bei match < 100 % = nur noch Relocations offen, die KI kann nicht mehr helfen.
perfect — match 100 % (byte-exakt) struct — struct 100 %, Rest Relocations partial — kompilierbarer Seed failed — kompiliert nie

Diese vier Kategorien bestimmen, in welches output/-Verzeichnis ein Ergebnis wandert und ob es als verwertbarer Seed für den externen Permuter taugt.

40_wrapper.py — die Hauptpipeline

Der Hauptwrapper verarbeitet ganze Verzeichnisbäume von C-Entwürfen parallel (ThreadPoolExecutor). Pro Funktion ruft er process_function auf, das die unten beschriebenen Phasen durchläuft. Fertige Funktionen (perfect/struct) werden vorab gefiltert, ein --resume-Mechanismus erlaubt das Fortsetzen abgebrochener Läufe.

4.1  Die Phasen

0
Phase 0 — Cluster-Transplant-Vorflug. Existiert ein bereits gelöstes ähnliches Funktion (perfect/struct) mit hoher ASM-Ähnlichkeit (DB-Score ≥ 90 %), wird dessen Referenz-Lösung mit den Symbolen dieser Funktion transplantiert und live verifiziert. Trifft es byte-exakt → sofort speichern, 0 KI-Calls. Fängt exakte ASM-Cluster ab.
A
Phase A — Syntax. Den Entwurf zum Kompilieren bringen. Zuerst deterministischer Auto-Fix (Redeclaration, falscher Rückgabetyp, undefinierte Symbole — aus den autoritativen Header-Typen). Erst wenn das nicht reicht, KI-syntax_fix-Calls (mit optionaler Thinking-/Batch- Eskalation). Scheitert alles, ein Rescue über die ähnliche Referenz.
B
Phase B/C — Diff-Minimierung. Iterative Schleife: kompilieren, evaluate_match, JSON-Diff erzeugen, KI gezielt ändern lassen. Fortschritt wird über struct_score getrackt; bei Stagnation steigt die Temperatur, dann wird der Modus eskaliert. Erreicht eine Runde match 100 % → perfect; struct 100 % → struct (Abbruch, da nur Relocations offen).
P
Permuter Fast-Lane. Sobald der Diff klein genug ist (is_permuter_ready), fährt der micro_permuter (Regex- + AST-Permutationen, Beam-Suche, similar-seeded) deterministisch dazwischen. Löst er es byte-exakt, ist die Funktion fertig — ebenfalls ohne weitere KI.
Konstanten (modus-abhängig) MAX_SYNTAX_RETRIES = 8 · MAX_DIFF_ROUNDS = 20 (thinking-artig) bzw. 50 (oneshot) · Temperatur 0.5 → 0.7 bei Stagnation · Abbruch nach 8/15 stagnierenden Runden bei Max-Temp. Jeder Erfolg wird in data/session/ persistiert → überlebt Neustarts.

4.2  Eskalations-Modi (CONV_MODE)

Die KI-Schnittstelle (modules/ai_agent.py) kennt mehrere Betriebsarten. Der Kern-Trick: teure Modi werden erst bei nachgewiesener Stagnation zugeschaltet, statt von Anfang an Tokens zu verbrennen.

ModusVerhalten
--oneshotJeder Call isoliert; History/Verlauf wird als Kontext in den Prompt geschrieben. Schnell, deterministisch-nah.
--thinkingQwen3.6-Multi-Turn mit Thinking Preservation. Reasoning wird lokal geloggt, aber nie in die nächste API-Nachricht zurückgespielt (verhindert Selbstkorrektur-Loops).
--dynamicStartet als oneshot, eskaliert einmalig zu einem Thinking-Call, wenn eine Funktion bei < 80 % struct festhängt.
--dynamic-batchStartet als oneshot, eskaliert bei Stagnation zu Best-of-N-Batch mit adaptivem Budget (Basis 8 + Bonus je Kontext-Stufe). Die Diversität kommt aus dem Kontext-Fahrplan, nicht aus Temp-Spread.
--dynamic-compilerOrthogonal: eskaliert Syntax-Fixes zu Thinking, wenn der Compile-Loop 5× in Folge scheitert. Mit jedem anderen Modus kombinierbar.
-b NGlobales Best-of-N: feuert N parallele Kandidaten pro Runde, der beste (nach autoritativem Score) gewinnt.

Welche Modelle/Backends hinter diesen Modi stehen — und warum lokal vs. Cloud aus Kostengründen getrennt sind — steht in §4.5.

4.3  Der Kontext-Fahrplan

Wenn eine Funktion mit Vollkontext stagniert, ist oft weniger Rauschen die richtige Antwort. Statt im Batch N fast identische Prompts zu feuern, variiert der Fahrplan die Kontext-Zusammensetzung über die Slots. Der ASM-Diff (einzige Wahrheit) und die harten TechEnv-Fakten (Namen, Größen, Stack-Layout, Calls) bleiben immer; reduziert werden nur inferierte Felder, das Cheatsheet und zuletzt die ähnliche Referenz.

StufeTechEnvCheatsheetSimilarZweck
fullvoll✓✓maximaler Kontext
no_csvoll✗✓Cheatsheet-Rauschen raus
hard_envnur harte Fakten✗✓inferierte Typen raus
diff_onlynur harte Fakten✗✗befreit von falschem Mikro-Muster

Stufen, die sich nur in fehlenden Zutaten unterscheiden, kollabieren automatisch — kein Slot wird auf einen identischen Prompt verschwendet. Das Batch-Budget wächst mit der Zahl wirklich unterscheidbarer Stufen.

4.4  No-AI Cluster-Scan (--no-ai)

0 Tokens, volle Geschwindigkeit Im --no-ai-Modus läuft die Pipeline ohne einen einzigen KI-Call: nur deterministischer Auto-Fix + Phase-0-Cluster-Transplantat + Beam-Permuter + Diff-Bewertung. Er überschreibt alle KI-Flags, erzwingt das Laden der Similar-DB und überspringt den Backend-Health-Check. Zweck: günstig abklopfen, wie viele Funktionen rein aus bestehenden ASM-Clustern fallen, bevor man teure KI ansetzt.

4.5  KI-Modelle & Backends (Kostenstrategie)

Die KI-Schnittstelle ist backend-agnostisch (OpenAI-kompatibel) und kennt zwei Betriebsarten, die bewusst aus Kostengründen getrennt sind:

Lokal — GPU-LLM über RunPod Ein selbst gehostetes Modell auf gemieteter GPU (RunPod), via vLLM OpenAI-kompatibel angebunden (LOCAL_API_BASE/_KEY/_MODEL). Der primäre Pfad für den Massenbetrieb — tausende Calls ohne Pro-Token-Abrechnung, volle Kontrolle über Sampling und Thinking.
Cloud-API — Claude & Alibaba DashScope Anthropic Claude und Alibaba DashScope (Qwen) als austauschbare API-Backends — nützlich für Stichproben, schwere Einzelfälle oder wenn keine GPU läuft. Ohne Setup-Aufwand, aber pro Token abgerechnet.
Erprobtes Optimum Nach Vergleich mehrerer Modelle und Modi liefert Qwen/Qwen3.6-35B-A3B im Oneshot-Modus in dieser Pipeline die besten Ergebnisse: schnell und effizient. Das passt zur Architektur — weil Determinismus, Permuter und Wissensbasis dem Modell viel Arbeit abnehmen, schlägt ein schneller, günstiger Oneshot-Call das teure Dauer-Reasoning. Teurere Modi (thinking/batch) werden nur bei nachgewiesener Stagnation zugeschaltet.

Die Modell-Familie (Qwen3 vs. Gemma4) wird automatisch am Namen erkannt und steuert, wie Thinking aktiviert, Reasoning extrahiert und Sampling gesetzt wird. Optionales HTTP-Streaming verhindert Cloudflare-524-Timeouts bei langen Thinking-Calls.

51_wrapper_mother.py — der Mutter-Wrapper

Wofür dieser Wrapper existiert Die Hauptpipeline (0_wrapper) erzielt ihre besten Ergebnisse auf kleinen Funktionen — je größer der Code, desto mehr Diff-Stellen koppeln aneinander und desto schwerer wird ein byte-exakter Treffer in einem Rutsch. Der Mutter-Wrapper löst genau dieses Skalierungsproblem: er zerlegt eine große Funktion kontextbewusst in kleinere, beherrschbare Häppchen („Arme“) und serviert sie dem KI-Übersetzer einzeln — jeder Arm mit seinem eigenen, fokussierten Byte-Ziel. Damit greift dieselbe Agenten-Schleife, die bei kleinen Funktionen funktioniert, auch für riesige Mütter.

Riesige Funktionen lassen sich kaum als Ganzes auf byte-exakt treiben. Dieser Wrapper baut auf dem bewiesenen Modell des Alt-Splitters auf (siehe dort für das Match-Modell, 82/82 Arme byte-exakt). Er teilt sich Compiler-Treiber, KI und Wissensbasis mit dem Hauptwrapper, hat aber ein eigenes PIC-Match-Backend und importiert die non-PIC-Match-Logik des Hauptwrappers bewusst nie.

Die zentrale Idee (vom Splitter) Arme niemals standalone kompilieren (das scheitert an der Registerallokation). Stattdessen die ganze Mutter kompilieren und die echten Bytes an den C-Arm-Grenzen herausschneiden. Sind alle Arme byte-exakt, ist die Mutter by construction bitgenau.
1
SEED. Die Mutter-.s durch m2c jagen → strukturierter C: Signatur, Frame, Kontrollfluss + Erstentwurf jedes Arms. ?-Typen werden zu s32 saniert. (mother_seed)
2
GATE. Nur wasserdichte Mütter (genestete ifs, 1 Block/Arm) werden bearbeitet; Loop-/Switch-Mütter werden markiert & übersprungen.
A
PHASE A. Die ganze Mutter zum Kompilieren bringen (PIC, -g3) — deterministischer Auto-Fix, dann KI-Syntax-Fix (read-only aus dem Hauptwrapper).
B
PHASE B — front-to-back. Arm 0 verfeinern bis die Region im neu kompilierten Mutter byte-exakt ist, dann locken und zu Arm 1 (Upstream bleibt eingefroren). Nutzt dieselbe dynamic_batch-Maschinerie wie der Hauptwrapper: Temperatur/Stagnation, Eskalation zu Batch, similar-seeded Whole-Mother-Permuter als Fast-Lane. (mother_match, mother_similar)
W
WRITE. mother.c, pro Arm eine .c, und ein manifest.json mit byte-exakten (C-Arm ↔ Ziel-Hex)-Paaren.

Zusätzlicher deterministischer Modus: --split-asm

Exportiert pro Arm eine eigenständige byte-exakte Ziel-.s (keine KI), gespiegelt in der Zielhierarchie. Diese Arm-.s fließen als neue Targets in build_similar_asm_db.py — so füttert der Mutter-Wrapper die Ähnlichkeits-Wissensbasis des Hauptwrappers.

6Die Wissensbasis

Die Pipeline wird mit jedem Lösungserfolg besser. Zwei selbst aufgebaute Datenquellen leiten künftige Übersetzungen:

Similar-ASM-DB build_similar_asm_db.py findet für jede Ziel-Funktion den strukturell ähnlichsten bereits gelösten C-Code (Hex-Skeleton-Vergleich, multiprocessing). Quellen: alle perfect_matches + handmatchende Referenzen aus data/refdata_original. Ergebnis: data/similar_asm_db.jsonl — aktuell 14 641 Einträge.
Pattern-Cheatsheet modules/cheatsheet.py destilliert aus den 3 146 perfect-match C-Dateien wiederkehrende Muster (Entrypoint-Slots, Overlay-API-Nutzung, bekannte Signaturen). Wird inkrementell bei jedem neuen Perfect Match aktualisiert und als Kontext an den KI-Prompt angehängt.

Zwei Wege, wie eine ähnliche Referenz hilft

Sobald eine Referenz den Ähnlichkeits-Schwellenwert (70 % struct) reißt, wird sie an zwei Stellen eingespeist — das ist der Kern des „Lernens während der Ausführung“:

WegWie
Similar-Injektion in den Hill-Climbing-Permuter Der micro_permuter ist eine deterministische Bergsteiger-Suche (Hill Climbing) über semantisch-äquivalente C-Varianten. Statt nur vom rohen Entwurf zu starten, bekommt er die ähnliche Lösung als Seed — er klettert also von einem bereits hohen Punkt los und findet den byte-exakten Gipfel viel schneller. Greift bereits in Phase 0 (Cluster-Transplantat) und in der Fast-Lane.
Prompt-Injektion (--similar) Dieselbe Referenz wird dem LLM als strukturelle Vorlage in den Prompt gelegt: „kopiere diese Code-Struktur und Typen“. Das Muster einer echten, bereits gematchten Funktion schlägt arithmetisches Raten. Der Kontext-Fahrplan (§4.3) kann sie pro Batch-Slot zu- und abschalten.
Die selbstverstärkende Schleife → kritische Masse perfect match → speist Cheatsheet + Similar-DB → die nächste ähnliche Funktion startet mit besserer Referenz (Permuter-Seed und Prompt-Vorlage) → löst oft schon in Phase 0 ohne KI → neuer perfect match. Je mehr Funktionen übersetzt sind, desto stärker profitiert die Pipeline. Ab einer kritischen Masse deckt die Wissensbasis so viele ASM-Cluster ab, dass sich neue, ähnliche Funktionen nahezu von selbst lösen — der Agent trainiert sich praktisch während des Laufs selbst.

8Benutzung

WSL-Pflicht Der IDO cc ist ein Linux-Binär und läuft nur in WSL. Vor dem Lauf das LLM-Backend konfigurieren (LOCAL_API_BASE, LOCAL_API_KEY, LOCAL_MODEL; siehe 0_docs.md) bzw. DASHSCOPE_API_KEY.

Hauptpipeline (0_wrapper)

# Standardlauf: oneshot, 1 Worker
python3 0_wrapper.py --input data/c_input --target-asm data/target_asm

# Mit Wissensbasis, adaptivem Batch, 8 Workern, Resume
python3 0_wrapper.py --cheatsheet --dynamic-batch --dynamic-compiler -j8 --resume

# Cluster-Scan ganz ohne KI (0 Tokens) — nur Determinismus + Permuter
python3 0_wrapper.py --no-ai -j8

# Live-Ansicht (erzwingt -j1)
python3 0_wrapper.py --thinking --streaming --live

Mutter-Wrapper (1_wrapper_mother)

# Mutter-.s nach data/target_asm/mother_functions/ legen, dann:
python3 1_wrapper_mother.py --similar --dynamic-batch

11Glossar

TermErklärung
Match / byte-exaktDer kompilierte C erzeugt exakt dieselben Maschinenbytes wie das Ziel-ROM. Das Endziel jeder Funktion.
match_rateAnteil byte-identischer Instruktionen inkl. korrekter Relocations (100 % = perfekt).
struct_scoreStruktureller Ähnlichkeitsscore, robust gegen Relocations; 100 % bei match < 100 % = nur Relocations offen.
RelocationPlatzhalter für noch nicht aufgelöste Adressen (%hi/%lo/jal); werden über Symbol+Typ verglichen, nicht über Hex.
PIC / non-PICPosition Independent Code. Der Mutter-Wrapper kompiliert PIC, der Hauptwrapper non-PIC (-non_shared); die Differenz wird im Vergleich überbrückt.
TechEnvPro-Funktion-Metadaten (JSON): Typen, Stack-Frame, Saved-Register, Calls, inferierte Argumente — teils harte Fakten, teils Inferenzen.
Cheatsheetaus perfect_matches destillierte wiederkehrende C-Muster, als Prompt-Kontext.
Similar-DBpro Ziel der strukturell ähnlichste gelöste C-Code — Quelle für Phase 0, Permuter-Seed und Prompt-Injektion.
Mutter / ArmMutter = große Zielfunktion; Arm = maximale Folge von Payload-Anweisungen ohne eigene Kontrollstruktur.
Thinking PreservationQwen3-Multi-Turn-Technik: Reasoning bleibt lokal, wird aber nie in die nächste API-Nachricht zurückgespielt.
Permuterprobiert semantisch gleichwertige C-Varianten durch, um Byte-Matches zu finden.
